# PR 差分レビュー

指定された PR の差分を「実装内容の妥当性」の観点でレビューする。
`AGENTS.md` / `docs/ai/ai-collaboration.md` の Codex の役割（レビュー担当）の実行版。

## 対象の特定と情報収集

1. PR 番号が指定されていれば `gh pr view <N>` で概要・本文・関連 Issue を確認し、`gh pr diff <N>` で差分を取得する
2. PR 番号がなければカレントブランチと `main` の差分（`git diff main...HEAD`）を対象にする
3. 関連 Issue（`Closes #N`）があれば読み、変更の意図・Acceptance Criteria を把握してからレビューする
4. PR / base 差分に `^supabase/migrations/[0-9]{14}_.+\.sql$` と一致するファイルがあれば、毎回 `migration-check` スキルを実行する

### Migration 差分がある場合

- `.agents/skills/migration-check/SKILL.md` を読み、対象 migration のローカル適用、SQL、型、docs を検証する
- 適用失敗、または対象 timestamp が Local の migration 履歴に存在しない場合は原則 P1 とする
- Docker / Supabase CLI が利用できず実行できない場合は成功扱いにしない。環境要因自体は指摘にせず、未確認の項目と理由を「補足」に記載する
- 「補足」には、対象ファイル、reset、migration list、SQL チェックリスト、型 / docs 同期の各結果を、`成功 / 失敗 / 未確認` のいずれかとともに記載する

## レビューしないこと

- Typecheck / Lint / Test / Build の結果（CI で確認するため対象外）
- コードスタイル・フォーマットなどの機械的なコード品質チェック
- 差分と無関係な既存コードへの指摘（将来の注意点として「補足」に書くのは可）

## レビュー観点

- **セキュリティ**: 認証・認可チェックの漏れ、RLS との整合（本人データのみ・`user_id` スコープ）、
  入力バリデーションの欠落（入力境界で必ず検証 — `AGENTS.md`）、secrets の混入、
  `SUPABASE_SERVICE_ROLE_KEY` のクライアント側流出、Storage の公開 URL 利用（signed URL 前提 — `docs/database.md`）
- **設計整合性**: `rules/architecture.md`（UI → UseCase → Repository の一方向依存、層責務）と
  ADR（`docs/decisions/`）への準拠。既存パターン（atomic RPC、participant 差分更新、signed URL 経由のメディア表示等）からの不要な逸脱
- **パフォーマンス**: N+1 クエリ、不要な join / payload、signed URL の過剰生成、再検証の漏れ・過剰
- **将来拡張性**: データモデルや型の設計が既知の今後の計画（`docs/roadmap.md`、open な Issue）を不必要に困難にしないか

## 出力形式（必ずこの形式にする）

```markdown
セルフレビュー結果（観点: セキュリティ / 設計整合性 / パフォーマンス / 将来拡張性）

## P0

なし

## P1

なし

## P2

なし

## 補足

なし
```

### 分類基準

- **P0**: マージ前に必ず修正すべき重大問題（セキュリティ欠陥、データ破損、明確なバグ）
- **P1**: マージ前の修正を強く推奨する問題
- **P2**: 今回の PR で直すと望ましい改善点
- **補足**: 問題ではないが確認したこと、妥当と判断した理由、将来の注意点

### 指摘の書き方

- 問題がない区分は「なし」と明記する
- 指摘は 1 件ごとに以下を簡潔に書く:
  - 対象: `ファイルパス:行` または関数名
  - 問題点と理由（P0/P1 は、なぜその優先度かが分かるように）
  - 修正方針（具体的に。ただし実装の強制はしない）
- 差分から判断できる内容に限定し、推測だけの指摘はしない
  （確認できないが気になる点は「補足」に確認事項として書く）
- 既存設計を尊重し、大規模リファクタリング提案は明確な必要がある場合のみ行う
- 設計レベルの議論が必要な指摘は、PR ではなく Issue 化を提案する（設計議論は Issue で行う — `docs/ai/ai-collaboration.md`）
- 出力は日本語ベース

## 出力先

- 既定では PR にコメントとして投稿する（`gh pr comment <N> --body-file <一時ファイル>`）。
  投稿後、投稿した内容の要約と PR コメントの URL を会話にも提示する
- 「会話だけでいい」「コメントしないで」など明示された場合のみ、投稿せず会話出力に留める
- カレントブランチ比較（PR が存在しない）場合は投稿先がないため会話に出力する
