# 坂道トーク帖 — Product Foundation

この文書は、「坂道トーク帖」におけるプロダクト設計、デザイン判断、Product Design Audit、Impeccable Critiqueの判断基準を定める。

本書は、プロダクトとして何を優先するかを扱う。UIの具体的な見た目、色、余白、タイポグラフィ、コンポーネント仕様などは扱わず、別途 `DESIGN.md` で定義する。

## Register

product

## Platform

web

## Product Overview

「坂道トーク帖」は、坂道メンバーから届いたトークを、元のサービスや受信環境から独立して長期的に残し、あとから思い出として振り返るための個人用Webアーカイブである。

対象となるトークは、乃木坂46、櫻坂46、日向坂46のメンバーまたはグループトークに属する。テキストだけでなく、画像、動画、音声と、その発言者や投稿日時を含む時間の記録を扱う。

リアルタイムに会話するためのサービスではない。過去に届いた言葉や時間を見つけ直し、前後の流れを読み、そのときの文脈へ戻るためのプロダクトである。

## Product Purpose

坂道メンバーから届いた言葉や時間を、元のサービスや受信環境に依存せず長期保存し、あとから継続的に再訪できるようにする。

中心価値は、単にデータを保管することや、管理作業を効率化することではない。自分にとって大切な言葉や思い出を、時間の流れとともに振り返れる状態をつくることである。

## Primary User

主な利用者は、坂道メンバーから届いたトークを個人的に残し、長期的に振り返りたい単一の認証ユーザーである。

利用は個人のためのものであり、他者との共同管理、公開、共有、交流を前提としない。ユーザー自身は会話の参加者として保持せず、トークを届けた相手側のメンバーを参加者として扱う。

## Core User Outcome

最も重要なユーザー成果は、次のとおりである。

> 過去に届いた言葉や時間を、思い出として気持ちよく振り返り、そのときの文脈へ戻れること。

目的の一件を見つけられるだけでは十分ではない。その前後に何が届いていたか、どのような時間の中にあったかを読み直し、記憶をたどれることまでを成果とする。

## Product Experience Priority

プロダクト体験は、次の優先順位で判断する。

1. 過去の言葉や時間を、思い出として落ち着いて振り返れること
2. 目的のトークを見つけ、前後の文脈へ自然に戻れること
3. テキストとメディアを、長期的な記録として継続して閲覧できること
4. アーカイブを正確に維持するための操作が、効率的かつ信頼できること

インポート、編集、メンテナンスは重要だが、中心体験を支える運用機能として位置づける。運用効率が、振り返りや文脈の再体験より上位の判断基準になることはない。

## Core Content Model

### 会話

トークを振り返る基本単位。会話には次の情報が属する。

- タイトル
- 乃木坂46、櫻坂46、日向坂46のいずれかのグループ
- 1人以上の相手側参加者
- 任意のトップ画像
- 1つ以上の会話期間
- 時系列に並ぶトークレコード

会話期間は複数の区間を持つことができ、トークを受け取っていなかった期間を区別する。

### 参加者

会話内でトークを届けた相手を表す。1人トークと複数人トークは参加者数によって区別され、各トークレコードの発言者として参照される。

### トークレコード

会話の中で届いた個々の内容を表す。次の種類を扱う。

- テキスト
- 画像
- 動画
- 音声

各レコードは発言者と投稿日時を持つ。種類に応じて、タイトル、本文、メディアを保持する。表示順は、当時の流れを再現できる時系列を基本とする。

### 添付メディア

画像、動画、音声の実体をトークレコードに結び付ける。メディアファイルがまだ用意できない場合でも記録を先に残し、あとから添付して完成させられる。

## Primary User Workflows

### 振り返る

1. 会話一覧または検索から、見返したいトークを探す
2. 対象の会話とトークを開く
3. 前後のタイムラインを読む
4. 当時の文脈と記憶へ戻る

### 会話から探す

1. グループを手掛かりに会話一覧を見る
2. メンバーやグループトークの会話を選ぶ
3. タイムライン、会話内検索、日付、メディアを手掛かりに目的の内容へ移動する

### 検索から探す

1. 記憶にある言葉を検索する
2. 該当するトークレコードを選ぶ
3. 元の会話内の位置へ移動する
4. 前後のタイムラインを読んで文脈を取り戻す

### トークを記録する

1. 会話と参加者、会話期間を用意する
2. テキスト、画像、動画、音声を投稿日時と発言者に結び付けて追加する
3. 必要に応じて未添付メディアをあとから補完する

### まとめて取り込む

1. 取り込み先の会話を選ぶ
2. JSONまたは `.eml` を読み込む
3. 件数、期間、内容の内訳、重複、エラー、参加者の割り当てを事前に確認する
4. インポートを実行し、結果を確認する
5. 必要に応じて未添付メディアや会話情報を補完する

### アーカイブを維持する

会話情報、参加者、期間、画像、トークレコード、表示名を必要に応じて更新し、アーカイブを継続して振り返れる状態に保つ。

## Rediscovery Model

主要な再訪モデルは、次のとおりである。

```text
検索または会話一覧
→ 目的のトークを見つける
→ そのトークを開く
→ 前後のタイムラインを読む
→ 当時の文脈と記憶へ戻る
```

検索と会話一覧は、大量のアーカイブから目的地を見つけるための入口である。

タイムラインは、見つけた一件を孤立したデータとして表示する場所ではない。時間の流れと前後関係を読み直し、思い出を再体験する中心である。検索結果、日付検索、メディア一覧などから移動した場合も、最終的には元のタイムラインの文脈へ戻れることを重視する。

## Device and Task Model

モバイルとデスクトップはどちらも重要であり、同じアーカイブへアクセスするための対等な利用環境として扱う。ただし、中心となるタスクは異なる。

### モバイル

主に振り返りと再発見を支える。

- タイムラインを読む
- 検索する
- 過去のコンテンツを見つける
- 画像、動画、音声を閲覧する
- 前後の文脈へ戻る

### デスクトップ

モバイルと同じ閲覧・再発見体験に加え、負荷の高い運用作業を支える。

- 一括インポート
- 編集
- アーカイブのメンテナンス
- 大量データの確認と管理

坂道トーク帖を、デスクトップ中心の管理アプリとして定義しない。どちらの端末でも中心となる振り返り体験が成立することを前提とする。

## Product Principles

### 1. 記憶を管理より先に置く

効率や情報量だけでなく、過去の言葉を思い出として振り返れるかを最優先に判断する。

### 2. 一件より文脈を大切にする

個々のトークを見つけることに加え、その前後の時間の流れへ戻れることを重視する。

### 3. 再発見から再体験までを一続きにする

会話一覧と検索を入口とし、最終的にタイムラインの文脈へ戻れる体験を守る。

### 4. 静かな個人記録帖である

落ち着き、個人的な親密さ、温かみを持つアーカイブ体験を目指す。データベース管理ツール、コレクションダッシュボード、SNS、コミュニティの性格を中心にしない。

### 5. 内容を主役にする

プロダクト自身の主張より、届いた言葉、画像、動画、音声と、それらを読む時間を優先する。

### 6. 元の環境から独立して残す

元のサービスや受信環境に依存せず、長期的に保存し、あとから再訪できることをプロダクトの継続的な目的とする。

### 7. 運用機能は中心体験を支える

インポート、編集、メンテナンスは、効率的かつ信頼できる必要がある。ただし、プロダクトの感情的な性格を定義する中心体験にはしない。

### 8. ひとつのプロダクトとして一貫させる

乃木坂46、櫻坂46、日向坂46の違いは情報として尊重するが、グループごとに別のプロダクト体験へ分断しない。

### 9. 端末ごとの役割を尊重する

モバイルでは閲覧と再発見、デスクトップではそれらに加えて運用と管理を支える。どちらか一方だけを基準にプロダクトを定義しない。

### 10. 利用環境が変わっても中心体験を保つ

ライト・ダーク双方を含む利用環境の違いによって、内容の読みやすさや振り返りの体験が損なわれないことを前提とする。

### 11. 未確定の将来要件で現在を複雑化しない

将来機能は、具体的な要件と価値が確認されてから設計する。未確定の可能性を理由に、現在の中心体験や情報構造を複雑化しない。

## Operational Capabilities

中心体験を成立させ、継続して利用するために、次の運用能力を持つ。

- 認証された単一ユーザーによる利用
- 会話、参加者、会話期間、トップ画像の作成と管理
- テキスト、画像、動画、音声の個別追加と閲覧
- トークレコードの編集と削除
- 未添付メディアの記録と事後添付
- 全文検索、会話内検索、日付検索
- 会話内メディア一覧
- JSON一括インポート
- `.eml` 一括インポート
- インポート前の内容確認、重複の識別、参加者の割り当て、結果確認
- 自分の名前をトーク内で置き換えて表示するための表示名設定

これらはアーカイブを成立・維持させる重要な機能だが、プロダクトの優先順位は常に振り返りと文脈への再訪を基準とする。

## Scope Boundaries

現在のプロダクトは、次の範囲を前提とする。

- 単一ユーザーによる個人利用
- 認証必須の非公開アーカイブ
- 乃木坂46、櫻坂46、日向坂46に属する会話
- テキスト、画像、動画、音声の保存と閲覧
- 会話と参加者、投稿日時を軸にした時系列記録
- 会話一覧と検索を入口とする再発見
- タイムラインを中心とする文脈の再体験
- 個別追加と一括インポートの両方による記録

## Non-Goals

現在のプロダクトでは、次を目的としない。

- リアルタイムチャットやメッセージ送信
- マルチユーザーによる共同利用・共同編集
- アーカイブの公開共有
- SNS機能
- コミュニティ機能
- 他者からの反応、評価、フォローを前提とする体験
- ファン活動の実績や収集量を競うコレクションダッシュボード
- 管理効率そのものを中心価値とするデータベース管理ツール

## Future Considerations

次の項目は将来検討事項であり、現在の確定要件ではない。

- PWA対応
- タグ・ラベル
- バックアップ／エクスポートのワークフロー

これらは、実際の利用上の課題と要件が明確になった時点で検討する。将来対応の可能性だけを理由に、現在のプロダクト原則、情報構造、中心体験を複雑化しない。

将来機能を追加する場合も、「過去に届いた言葉や時間を、思い出として気持ちよく振り返り、そのときの文脈へ戻れること」を最上位の判断基準とする。
