# 現状調査（リファクタリング監査）

talk-archive を観点別に調査し、`docs/audits/` に監査ドキュメントを残す。
（`docs/audits/` が無ければ初回に作成する）

## 手順

### 1. 前回監査の確認

- `docs/audits/` の過去監査を読み、前回の指摘・Issue 化状況を把握する
- 2回目以降の監査では、指摘ごとに **改善済み / 残存 / 新規** を区別して報告する
  （前回と同じ内容を新規発見のように書かない）

### 2. 観点別調査（優先順位順）

1. **セキュリティ（認証・認可含む）**: 認証ガードの範囲、RLS ポリシー（`user_id` スコープの一貫性）、
   Server Actions の認証チェック、`SUPABASE_SERVICE_ROLE_KEY` の利用箇所（サーバー専用の原則）、
   Storage の signed URL 運用（公開 URL を使っていないか）、
   XSS ベクター（`dangerouslySetInnerHTML`、URL スキーム検証）、secrets の扱い
2. **データアクセス**: repository パターンの一貫性、エラーハンドリング、N+1、
   型安全性（キャスト・手書き Row 型の量、`src/types/database.ts` 生成型の利用）、
   トランザクション境界（atomic RPC のパターン）
3. **アーキテクチャ・共通化**: 層の依存方向違反（`rules/architecture.md`）、
   巨大ファイル（`wc -l` 上位）、ファイル間の重複、god module 化の兆候
4. **UI設計・状態管理**: client/server コンポーネント境界、フォームの状態管理パターンの重複、
   エラーバウンダリ・404 の整備状況
5. **パフォーマンス**: キャッシュ・再検証戦略、全件取得のスケール限界（レコード数増加時）、
   `next/image`、signed URL 生成のコスト、バンドルへの影響
6. **横断**: テストカバレッジの偏り（Vitest）、依存パッケージ健全性、typecheck / lint / test の通過

### 3. 調査の規律

- 指摘には必ず根拠を付ける: `ファイルパス:行` / migration ファイル名 / 実行したコマンドの結果
- ADR（`docs/decisions/`）・Issue で**意図的に決定済み**の事項は、欠陥ではなく「決定とその前提」として扱う
- コードから確認できないこと（Supabase ダッシュボード設定等）は「要確認」として明示し、
  勝手に問題と断定しない
- 良い点も記録する（次回監査の基準線になる）

### 4. 監査ドキュメントの作成

- `docs/audits/NNN-current-state-audit.md`（連番は audits 内の最大 + 1、初回は 001）
- 構成: 作成日・対象・調査観点 → 総評 → 観点別（良い点 / 指摘事項）→
  **優先順位付き改善計画表**（P1/P2/P3、規模・効果付き）→ Issue 化状況
- 日本語ベースで記載する

### 5. Issue 化とドキュメント反映

- 改善項目の Issue 化を提案し、承認されたら `/design-issue` の手順で起票する
- 中長期の課題は `docs/roadmap.md` の該当セクションに反映する
- 前回監査の指摘が解消済みなら、該当ドキュメントを更新する（`/doc-sync` 相当）

## 完了条件

- 監査ドキュメントが根拠付き・優先順位付きで作成されている
- 前回監査との差分（改善済み / 残存 / 新規）が明示されている（2回目以降）
- Issue 化・roadmap 反映が完了しているか、提案として提示されている
