# 坂道トーク帖 要件定義

## 概要

「坂道トーク帖」は、坂道メンバーから届いた言葉と時間を残す。私だけの記録帖。
外部アプリなどで受け取ったトークを保存・閲覧するための個人用Webアプリとする。

リアルタイムチャットではなく、**トーク履歴の保存と検索を目的とする。**

---

## 利用者

- 単一ユーザー
- 認証必須
- 個人利用のみ

---

## 会話メタデータ

- 各会話は **乃木坂 / 櫻坂 / 日向坂** のいずれか 1 グループに属する
- グループトークも必ずいずれか 1 グループに属する
- 各会話は相手側の参加者を 1 人以上持つ
  - 自分自身は参加者として保持しない
  - 1 人トークか複数人トークかは参加者数で判断する
  - participant は record の発言者参照に使うため、record 作成後の削除は許可しない
  - participant 編集は「名前変更」と「追加」を前提にし、既存 participant の ID を維持する
- 各会話はトップ画像を持てる
  - トップ画像は任意
  - 未設定時はプレースホルダー画像を表示する
- 各会話は会話期間を持つ
  - 会話期間は 1 つ以上の区間で管理する
  - 区間は開始日必須、終了日は任意
  - 終了日未入力は「現在進行中」を表す
  - 会話期間の日数は区間の和集合を日単位で数える
  - 区間の間にトークを受け取れないギャップがあれば、その日数は含めない
  - 現在進行中の区間は今日までの日数を表示する
  - 実際にトークが送られてきた日数とは別概念とする

---

## 保存するトーク形式

### テキスト

- タイトル（任意）
- テキスト（必須）
- 投稿日時（必須）
- 発言者（必須）
- 投稿日時は JST 基準で入力・表示する

### 画像

- タイトル（任意）
- テキスト（任意）
- 画像ファイル（1枚。未添付保存する場合を除き必須）
- 投稿日時（必須）
- 発言者（必須）
- 投稿日時は JST 基準で入力・表示する
- 詳細ページでインライン表示する

### 動画

- 動画ファイル（未添付保存する場合を除き必須）
- 音声あり / なし
- 投稿日時（必須）
- 発言者（必須）
- 投稿日時は JST 基準で入力・表示する
- 詳細ページで HTML5 プレーヤー再生できる

### 音声

- 音声ファイル（未添付保存する場合を除き必須）
- 投稿日時（必須）
- 発言者（必須）
- 投稿日時は JST 基準で入力・表示する
- 詳細ページで HTML5 プレーヤー再生できる

### メディア未添付レコード（#113）

- 画像 / 動画 / 音声は、composer で「ファイルはあとで添付する」を選んだ場合のみ
  メディアファイルなし（種別・投稿日時・発言者のみ）で保存できる
- 未添付かどうかは attachment の有無で判定する（状態カラムは持たない）
- 未添付レコードはタイムライン上で「◯◯未添付」バッジ表示され、その場でファイルを添付できる
- 添付ファイルの種別・サイズ制限は通常のメディアレコードと同じ（画像 10MB / 動画・音声 50MB）

---

## トーク詳細画面

- トーク詳細画面では、会話メタデータ本文ではなくトークのやり取りを主に表示する
- 上部に固定ヘッダーを持つ
  - トークタイトル
  - 会話内検索ボタン
  - 三点リーダ
- 三点リーダから以下へ遷移できる
  - 概要
  - 日付検索
  - 会話内メディア一覧
  - 会話編集（詳細画面内の編集モード切替）
  - 日付検索は詳細画面上のモーダルで開く
  - 会話メタデータ編集は概要画面から行う
- タイムラインは LINE 風の吹き出しレイアウトにする
  - participant 名とイニシャル丸アイコンを表示する
  - 各吹き出しに投稿日時を表示する
  - レコードの編集・削除は編集モード中のみ操作できる
- トーク追加は画面下部のタイプ切替式 composer で行う
  - 先にタイプを選び、そのタイプに必要な入力だけ表示する
  - participant が 1 人の会話では speaker 選択を省略して自動設定する
- 会話内検索は全文検索で、絞り込みではなく該当位置へジャンプする
- 日付検索はモーダルを閉じるだけで詳細画面へ戻れる
- 会話内メディア一覧からは詳細画面へ戻りやすい導線を持つ

## 主な機能

- 会話管理
  - グループ切替（乃木坂 / 櫻坂 / 日向坂）
  - カード形式のトーク一覧
  - カードにトップ画像、トーク名、会話期間日数を表示
- トーク保存
- メディア保存
- メディア表示
  - 画像のインライン表示
  - 動画の HTML5 プレーヤー表示
  - 音声の HTML5 プレーヤー表示
  - 非公開 Storage には signed URL でアクセスする
- 全文検索
- 会話内メディア一覧
  - 画像 / 動画 / 音声を会話ごとに分けて表示する
- トークJSON一括インポート（Phase 6 / #114）
  - インポート前にプレビュー画面で総件数・取り込み対象件数・重複スキップ件数・期間・種別内訳・行エラーを確認できる
  - 既存レコードとの重複（participant + 投稿日時 + 種別 + 本文先頭20文字）は自動でスキップする（判定・participant解決は会話行ロック内で行い、並行実行時の重複作成を防ぐ）
  - メディア未添付のレコード（ファイルはあとから添付）としてまとめて取り込める
  - 1回のインポートで扱えるのはファイルサイズ5MB・5,000件まで（超える場合はファイルを分割する）
- .emlインポート（Phase 6 / #115、#128 で参加者割り当てを簡素化、#129 で実メールの構造に対応、#133 で複数画像を全件登録するよう変更）
  - JSON インポートと同じ画面でモードを切り替え、.eml ファイル（複数）を選択してプレビュー・一括保存できる
  - 本文・受信日時（JST）を自動抽出する
  - メールのトークは常に1対1のため、差出人（From アドレス）ごとの割り当ては行わない。取り込み先トークの参加者へ自動割り当てする
    - 参加者が1人の場合、全メールをその参加者へ自動で割り当てる（割り当て先を画面に表示）
    - 参加者が複数の場合、バッチ全体で割り当てる1人を選択する（既定値は先頭の参加者）
    - 参加者が0人の場合は .eml インポートを実行できない（先に参加者を追加するよう案内する）
  - 選択した .eml 群に2種類以上の From アドレスが含まれる場合、別の人のメールが混ざっていないか確認を促す警告をプレビューに表示する（取り込み自体はブロックしない）
  - 本文抽出（#129: 実メールの text/plain には配信システム由来の定型スタブや Gmail 由来のマーカー行が混在するため）
    - text/plain 本文からは Gmail アプリが画像添付を示す `[image: ...]` 形式のマーカー行のみを除去し、除去でできた連続空行は詰める
    - text/plain 本文中の数値文字参照（`&#NNNN;` / `&#xHHHH;`）はデコードする（名前付きエンティティ `&amp;` 等は text/plain ではリテラルの可能性があるためデコードしない）
    - text/plain が上記処理後に空、または「メールがうまく表示されない方はこちらをご覧ください」で始まる配信システムの定型スタブのみの場合は、HTML パートから抽出した本文を採用する
  - 画像は添付ファイルとして受け取る場合と、HTML 本文内の `<img src="https://...">` によるリモート画像参照として受け取る場合の両方に対応する（#129: 実メールは添付ファイルを持たず、画像は HTML パート内のリモート参照のみを持つ）
    - 画像付きメールは、含まれる画像を全件レコードとして取り込む（#133）。1枚目の画像は本文を保持したまま元のレコード（type: image）に添付し、2枚目以降はタイトルを引き継いだ独立した画像レコード（本文なし、投稿日時・発言者は元メールと同じ）として作成する。添付ファイルとリモート画像参照の両方がある場合は添付ファイルを優先し、HTML 内の画像参照はすべて無視する
    - リモート画像参照のみの場合はインポート実行時に全件を取得して添付する。取得先は SSRF・トラッキングピクセル対策として許可リスト方式に限定する（現状は `mail-web.c-nogizaka46.com` の `/mail/output/qimage` のみ。https・userinfo/ポート指定なし・リダイレクト拒否を強制し、許可外の `<img>` は取得対象にも画像件数にも含めない）
    - リモート画像の取得は1枚あたり10MB（読み込み中に上限を強制）・バッチ合計50MB・タイムアウト15秒・同時取得数5に制限する（同時取得数5のため、バッチ合計の上限判定が効くまでの間に進行中の並列取得分だけ実際のダウンロード量が一時的に上限を最大約40MB超過し得る）
  - 画像添付に失敗した場合（添付処理自体の失敗、リモート画像取得の失敗・タイムアウト・非画像コンテンツ・サイズ超過を含む）はメディア未添付レコードとして取り込み、実行結果画面から添付し直せる
  - 1回のインポートで扱えるのは.emlファイル1通あたり10MB・一度に200通まで、かつ合計50MBまで（超える場合はファイルを分割する）。加えて、展開後のレコード件数（1メールが複数画像で複数レコードに展開された後の件数）が5,000件を超える場合もインポートを実行できない（超える場合はメール数または画像数を減らす）
  - .eml インポート由来のレコードは、安定した重複排除キー `import_key`（`msgid:<正規化（大文字小文字は保持。Message-ID の比較は大文字小文字を区別するため）したMessage-IDのSHA-256ハッシュ>` または、Message-ID が無い場合は生EMLバイト列のSHA-256ハッシュ `sha256:<ハッシュ値>`、いずれも末尾に `#<メール内での連番>` を付与）で重複を判定する（#133、キー導出の見直しは #134）。生の Message-ID 値そのものは保存せず、常に固定長のハッシュ値を格納する。ファイル名は使わない。ファイル名ベースだと同名だが別内容のメールが衝突して2件目が取り込まれなかったり、ファイルをリネームして再インポートすると重複と判定されなくなったりするため。同一 .eml ファイルから展開された複数レコード（メイン+追加画像）は participant / 投稿日時 / 種別が一致し得るため、本文プレフィックスベースの判定ではなく import_key の一致のみで重複を判定する。JSON インポート由来のレコードは import_key を持たず、従来どおり本文プレフィックスベースで判定する。プレビュー画面の重複判定も、既存レコードが import_key を持つ場合はそれを使って判定するため、実行結果（RPC のスキップ件数）と一致する
